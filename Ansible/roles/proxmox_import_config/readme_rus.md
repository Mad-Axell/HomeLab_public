## Обзор

Роль `proxmox_import_config` предназначена для импорта и валидации конфигурационных переменных для развертывания LXC контейнеров Proxmox. Она обеспечивает централизованный способ управления секретами и конфигурациями конкретных хостов, гарантируя целостность данных и безопасность.

### Назначение

- Импорт секретов из централизованного хранилища (`secrets.yaml`)
- Загрузка переменных конкретного хоста из `host_vars/`
- Валидация всех параметров перед продолжением работы
- Проверка существования хоста в инвентори
- Предоставление подробного отладочного вывода
- Корректная обработка ошибок с детальными сообщениями
- Выполнение предварительных проверок совместимости системы
- Отслеживание метрик выполнения и временных меток

## Возможности

- Автоматическая валидация с использованием `meta/argument_specs.yml` (Ansible 2.14+)
- Проверка типов для всех входных параметров
- Обеспечение обязательных параметров
- Валидация значений по умолчанию

### ✓ Предварительные проверки (НОВОЕ в v2.0)

- Проверка версии Ansible (>= 2.14)
- Проверка версии Python (>= 3.8)
- Проверка совместимости системы
- Раннее обнаружение проблем

- Валидация обязательных параметров

- Проверка наличия хоста в инвентори
- Проверка читаемости файлов
- Настраиваемый режим строгой валидации

- Пароли скрыты в отладочном выводе по умолчанию
- Отдельный режим отладки для чувствительных данных (`debug_sensitive`)

- Финальная сводка конфигурации
- Индикаторы статуса (✓/✗) для удобства чтения
- Метрики выполнения и временные метки

### ✓ Обработка ошибок

- Структура block/rescue для критических операций
- Детальные сообщения об ошибках с подсказками по устранению

- Отдельные сообщения об ошибках для каждого языка
- Простая отладка с подробными логами

### ✓ Гибкая конфигурация

- Множественные режимы отладки и языки
- Опциональная валидация
- Настраиваемые пути
- Управление выполнением на основе тегов
- Опции настройки производительности

## Требования

### Версия Ansible

- **Ansible >= 2.14** (требуется для автоматической валидации аргументов)

### Версия Python

- **Python >= 3.8**

### Системные требования

- Окружение Proxmox VE (для фактического развертывания)
- Доступ к файлу инвентори
- Доступ на чтение к файлам secrets и host_vars

### Зависимости от файлов

- Файл `secrets.yaml` с зашифрованными или открытыми учетными данными
- Файл YAML для конкретного хоста в директории `host_vars/`
- Файл инвентори с определением целевого хоста

## Переменные роли

### Настройки отладки и валидации

| Переменная | Тип | По умолчанию | Описание |
|------------|-----|--------------|----------|
| `debug_mode` | bool | `false` | Включить отладочный вывод для всех операций |
| `debug_lang` | string | `'both'` | Язык отладки: 'english', 'russian' или 'both' |
| `debug_sensitive` | bool | `false` | Включить отладочный вывод с паролями в открытом виде (НЕБЕЗОПАСНО) |
| `backup_enabled` | bool | `true` | Включить резервное копирование конфигураций (зарезервировано для будущего использования) |
| `validate_parameters` | bool | `true` | Включить валидацию параметров перед выполнением |
| `strict_validation` | bool | `true` | Включить режим строгой валидации со всеми проверками |

### Конфигурация хоста

| Переменная | Тип | По умолчанию | Описание |
|------------|-----|--------------|----------|
| `host_name` | string | `"some-server"` | Имя хоста для загрузки файла host_vars (ОБЯЗАТЕЛЬНО) |
| `host_vars` | string | `"some_server"` | Префикс для переменных в secrets.yaml (ОБЯЗАТЕЛЬНО) |

### Конфигурация путей

| Переменная | Тип | По умолчанию | Описание |
|------------|-----|--------------|----------|
| `ansible_base_dir` | string | `"/etc/ansible"` | Базовая директория для конфигурации Ansible |
| `ansible_host_vars_dir` | string | `"{{ ansible_base_dir }}/host_vars"` | Директория для переменных хоста |
| `ansible_vars_dir` | string | `"{{ ansible_base_dir }}/VARS"` | Директория для секретов и vault-файлов |
| `secrets_file_path` | string | `"{{ ansible_vars_dir }}/secrets.yaml"` | Путь к файлу секретов |
| `ansible_inventory_file` | string | `"{{ ansible_base_dir }}/hosts.yml"` | Путь к файлу инвентори |

### Настройки производительности и отката (НОВОЕ в v2.0)

| Переменная | Тип | По умолчанию | Описание |
|------------|-----|--------------|----------|
| `enable_rollback` | bool | `true` | Включить автоматический откат при ошибке |
| `async_timeout` | int | `300` | Таймаут асинхронных задач в секундах |
| `retries` | int | `3` | Количество повторов для неудачных задач |
| `retry_delay` | int | `5` | Задержка между повторами в секундах |

### Переменные, устанавливаемые ролью

Эти переменные устанавливаются ролью после импорта конфигураций:

#### Переменные сервера Proxmox

- `pve_node` - Имя узла Proxmox
- `pve_api_host` - Адрес хоста Proxmox API
- `pve_api_user` - Имя пользователя Proxmox API
- `pve_api_password` - Пароль Proxmox API

#### Переменные LXC контейнера

- `pve_lxc_root_password` - Пароль root для LXC контейнера
- `pve_lxc_root_authorized_pubkey` - SSH публичный ключ для пользователя root
- `pve_lxc_name` - Имя LXC контейнера
- `pve_lxc_vmid` - VMID LXC контейнера
- `pve_lxc_ostemplate_name` - Имя шаблона ОС
- `pve_lxc_ip_address` - IP адрес контейнера
- `pve_lxc_ip_mask` - Сетевая маска
- `pve_lxc_ip_gateway` - Шлюз по умолчанию
- `pve_lxc_nameserver` - DNS сервер имен
- `pve_lxc_searchdomain` - DNS домен поиска
- `pve_lxc_description` - Описание контейнера

#### Переменные администратора

- `admin_name` - Имя пользователя администратора
- `admin_password` - Пароль администратора

## Примеры использования

См. [example-playbook.yml](example-playbook.yml) для детальных примеров.

### Базовое использование

```yaml
- hosts: localhost
  roles:
    - role: proxmox_import_config
      vars:
        host_name: "web-server-01"
        host_vars: "web_server_01"
```

### С отладочным выводом (только русский)

```yaml
- hosts: localhost
  roles:
    - role: proxmox_import_config
      vars:
        host_name: "web-server-01"
        host_vars: "web_server_01"
        debug_mode: true
        debug_lang: 'russian'
```

### С пользовательскими путями

```yaml
- hosts: localhost
  roles:
    - role: proxmox_import_config
      vars:
        host_name: "db-server-01"
        host_vars: "db_server_01"
        ansible_base_dir: "/opt/ansible"
        secrets_file_path: "/opt/ansible/secrets/prod.yaml"
```

### С режимом отладки чувствительных данных (только для разработки)

```yaml
- hosts: localhost
  roles:
    - role: proxmox_import_config
      vars:
        host_name: "test-server"
        host_vars: "test_server"
        debug_mode: true
        debug_lang: 'russian'
        debug_sensitive: true  # ВНИМАНИЕ: Показывает пароли!
```

### Без валидации (не рекомендуется)

```yaml
- hosts: localhost
  roles:
    - role: proxmox_import_config
      vars:
        host_name: "quick-test"
        host_vars: "quick_test"
        validate_parameters: false
```

## Процесс валидации

Роль выполняет комплексную валидацию перед импортом конфигураций:

### 0. Автоматическая валидация аргументов (НОВОЕ в v2.0)

- Валидация всех входных параметров согласно `meta/argument_specs.yml`
- Проверка типов для всех переменных
- Обеспечение обязательных параметров
- Ранний выход при невалидных параметрах

### 1. Предварительные проверки (НОВОЕ в v2.0)

- Ansible версия >= 2.14
- Python версия >= 3.8
- Проверка совместимости системы

### 2. Проверка обязательных параметров

- Валидирует, что `host_name` и `host_vars` определены
- Проверяет, что значения параметров не пустые

### 3. Валидация файла секретов

- Проверяет существование файла по указанному пути
- Проверяет читаемость файла
- Проверяет размер файла

### 4. Валидация файла переменных хоста

- Проверяет существование `host_vars/<host_name>.yml`
- Проверяет читаемость файла
- Проверяет размер файла

### 5. Валидация файла инвентори

- Проверяет существование файла инвентори
- Проверяет читаемость файла

### 6. Проверка хоста в инвентори

- Ищет хост в файле инвентори
- Валидирует формат записи хоста
- Подтверждает настройку хоста

### Сводка валидации

После всех проверок отображается подробная сводка, показывающая:
- Статус каждой проверки валидации (✓/✗)
- Режим валидации (STRICT/NORMAL)
- Общий результат валидации

## Безопасность

### Защита чувствительных данных

1. **Защита no_log**: Все задачи, обрабатывающие пароли, используют `no_log: true` для предотвращения логирования чувствительных данных

2. **Скрытый вывод по умолчанию**: Пароли заменяются на `***HIDDEN***` в отладочном выводе по умолчанию

3. **Отдельный режим для чувствительных данных**: Используйте `debug_sensitive: true` только когда это абсолютно необходимо

4. **Обработка SSH ключей**: Публичные ключи загружаются безопасно из файлов

### Лучшие практики

- Держите `debug_sensitive: false` в продакшене
- Используйте Ansible Vault для `secrets.yaml`
- Ограничьте права доступа к чувствительным файлам:
  ```bash
  chmod 600 /etc/ansible/VARS/secrets.yaml
  chmod 600 /etc/ansible/host_vars/*.yml
  ```
- Используйте выделенные сервисные учетные записи с минимальными правами
- Регулярно меняйте пароли и API токены

## Режимы отладки

### Стандартный режим отладки (`debug_mode: true`)

Показывает:
- Шаги выполнения
- Пути конфигурации
- Статус переменных (пароли скрыты)
- Результаты валидации
- Финальную сводку
- Метрики выполнения (длительность, временная метка)

Пароли отображаются как: `***HIDDEN***`

### Выбор языка (`debug_lang`)

Выберите язык отладочного вывода:
- `'english'` - Только английский
- `'russian'` - Только русский
- `'both'` - Оба языка (по умолчанию)

### Режим отладки чувствительных данных (`debug_sensitive: true`)

**ВНИМАНИЕ**: Используйте только в безопасных окружениях!

Показывает все из стандартного режима ПЛЮС:
- Фактические значения паролей
- API учетные данные в открытом виде
- Все чувствительные данные конфигурации

Используйте только для:
- Устранения неполадок в изолированных окружениях
- Проверки начальной настройки
- Разработки (никогда в продакшене)

## Обработка ошибок

Роль использует структуру block/rescue для надежной обработки ошибок:

### Ошибка импорта секретов

Если импорт файла секретов не удался:
- Отображается детальное сообщение об ошибке на выбранном языке (языках)
- Предоставляются подсказки по устранению:
  - Проверьте существование и читаемость файла
  - Проверьте синтаксис YAML
  - Проверьте наличие требуемых переменных
- Выполнение роли завершается с описательным сообщением

### Ошибка импорта переменных хоста

Если импорт файла host_vars не удался:
- Отображается детальное сообщение об ошибке на выбранном языке (языках)
- Предоставляются подсказки по устранению:
  - Проверьте существование и читаемость файла
  - Проверьте синтаксис YAML
  - Проверьте правильность имени хоста
- Выполнение роли завершается с описательным сообщением

## Теги

Роль поддерживает следующие теги для гибкого выполнения:

### Основные теги

- `always` - Задачи, которые всегда выполняются (включая preflight и validation)
- `validate` - Задача валидации аргументов
- `preflight` - Проверки совместимости (НОВОЕ в v2.0)
- `validation` - Задачи валидации параметров и файлов
- `import` - Все операции импорта
- `debug` - Задачи отладочного вывода
- `summary` - Вывод финальной сводки

### Специфические теги

- `secrets` - Задачи, связанные с секретами
- `host_vars` - Задачи переменных хоста
- `set_facts` - Задачи установки переменных
- `files` - Операции с файлами
- `inventory` - Задачи, связанные с инвентори
- `sensitive` - Задачи с чувствительными данными
- `error` - Задачи обработки ошибок

### Примеры использования

Запустить только валидацию:
```bash
ansible-playbook playbook.yml --tags validation
```

Пропустить предварительные проверки:
```bash
ansible-playbook playbook.yml --skip-tags preflight
```

Запустить только импорт без отладки:
```bash
ansible-playbook playbook.yml --tags import --skip-tags debug
```

## Устранение неполадок

### Проблема: Слишком старая версия Ansible

**Ошибка**: `This role requires Ansible 2.14 or higher`

**Решения**:
1. Обновите Ansible: `pip install --upgrade ansible-core`
2. Проверьте текущую версию: `ansible --version`
3. Убедитесь, что у вас установлен как минимум Ansible 2.14

### Проблема: Слишком старая версия Python

**Ошибка**: `Python 3.8+ is required`

**Решения**:
1. Обновите Python до версии 3.8 или выше
2. Используйте виртуальное окружение с правильной версией Python
3. Обновите системную установку Python

### Проблема: Файл секретов не найден

**Ошибка**: `Secrets file does not exist or is not readable`

**Решения**:
1. Проверьте путь к файлу: `ls -la /etc/ansible/VARS/secrets.yaml`
2. Проверьте права доступа: `chmod 600 /etc/ansible/VARS/secrets.yaml`
3. Проверьте путь в переменной `secrets_file_path`

### Проблема: Файл переменных хоста не найден

**Ошибка**: `Host variables file does not exist or is not readable`

**Решения**:
1. Проверьте соответствие имени файла: `/etc/ansible/host_vars/<host_name>.yml`
2. Проверьте правильность переменной `host_name`
3. Проверьте права доступа к файлу

### Проблема: Хост не найден в инвентори

**Ошибка**: `Host 'xxx' not found in inventory file`

**Решения**:
1. Проверьте наличие хоста в инвентори: `grep <host_name> /etc/ansible/hosts.yml`
2. Проверьте правильность написания имени хоста
3. Убедитесь, что путь к файлу инвентори правильный

### Проблема: Переменная не найдена в секретах

**Ошибка**: Variable `xxx_pve_node` is undefined

**Решения**:
1. Проверьте, что префикс `host_vars` соответствует переменным в secrets.yaml
2. Проверьте наличие всех требуемых переменных:
   - `<prefix>_pve_node`
   - `<prefix>_pve_api_user`
   - `<prefix>_pve_api_password`
   - `<prefix>_pve_lxc_root_password`
   - `<prefix>_pve_lxc_root_authorized_pubkey`
   - `<prefix>_admin_user`
   - `<prefix>_admin_password`

### Проблема: Ошибка синтаксиса YAML

**Ошибка**: `YAML syntax error in file`

**Решения**:
1. Проверьте синтаксис YAML: `yamllint <file>`
2. Проверьте правильность отступов
3. Убедитесь в сбалансированности кавычек
4. Проверьте отсутствие табуляций (используйте пробелы)

### Проблема: Валидация аргументов не прошла

**Ошибка**: `validate_argument_spec failed`

**Решения**:
1. Проверьте, что все обязательные переменные предоставлены
2. Проверьте, что типы переменных соответствуют спецификациям
3. Просмотрите `meta/argument_specs.yml` для требований
4. Убедитесь, что значения переменных в пределах разрешенных диапазонов/выборов
