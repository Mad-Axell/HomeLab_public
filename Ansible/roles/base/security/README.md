# Security Role

Роль для базовой настройки безопасности операционной системы Linux.

## Описание

Эта роль реализует комплексные меры по защите операционной системы, включая:
- Управление правами доступа к файловой системе
- Защиту ядра Linux
- Защиту пользовательского пространства
- Сокращение атакуемой поверхности
- Общую защиту ОС

## Переменные

### Основные настройки
- `debug_mode` - включение режима отладки (по умолчанию: `True`)
- `ct_mode` - режим контейнера (по умолчанию: `True`)
- `locale` - локаль системы (по умолчанию: `en_US.UTF-8`)

### SSH настройки
- `allowed_ssh_subnet` - разрешенные подсети для SSH доступа
- `ssh_port` - порт SSH (по умолчанию: `22`)

### PAM настройки
- `pam_faillock_deny` - количество неудачных попыток входа
- `pam_unlock_time` - время блокировки в секундах
- `pam_pwquality_*` - параметры качества паролей

### Fail2ban настройки
- `fail2ban_bantime` - время бана в секундах
- `fail2ban_findtime` - окно поиска попыток
- `fail2ban_maxretry` - максимальное количество попыток

### Программное обеспечение
- `apt_to_add` - список пакетов для установки
- `ufw_rules` - правила файрвола UFW

## Задачи

### main.yaml
Основной файл роли, который включает все остальные задачи с учетом режима контейнера.

### access_rights_to_file_system_objects.yaml
**2.3.1** - Установка прав доступа к критическим системным файлам (`/etc/passwd`, `/etc/group`, `/etc/shadow`)
**2.3.2** - Защита исполняемых файлов в `/proc` (только для физических серверов)
**2.3.3** - Установка прав доступа к файлам cron
**2.3.4** - Поиск и защита файлов в переменной PATH
**2.3.5** - Установка прав доступа к скриптам автозапуска
**2.3.6** - Установка прав доступа к файлам cron
**2.3.7** - Установка прав доступа к пользовательским файлам cron
**2.3.8** - Установка прав доступа к исполняемым файлам и библиотекам ОС
**2.3.9** - Удаление прав на запись для файлов с setuid/setgid
**2.3.10** - Установка прав доступа к содержимому домашних директорий
**2.3.11** - Установка прав доступа к домашним директориям пользователей

### user_space_protection.yaml
**2.6.1** - Запрет подключения к другим процессам с помощью ptrace
**2.6.2** - Ограничение небезопасных опций для символических ссылок
**2.6.3** - Ограничение небезопасных опций для жестких ссылок
**2.6.4** - Защита от непреднамеренной записи в FIFO объект
**2.6.5** - Защита от непреднамеренной записи в файл
**2.6.6** - Запрет создания core dump

### harden_os.yaml
- Включение Address Space Layout Randomization
- Защита от загрузки модулей ядра
- Настройка auditd для мониторинга chmod
- Установка правильных прав доступа к `/etc/shadow`
- Отключение входа root через SSH
- Отключение пустых паролей в SSH
- Ограничение доступа SSH по IP подсети

### linux-kernel_protection.yaml
**2.4.1** - Ограничение доступа к логам ядра
**2.4.2** - Настройка kernel.kptr_restrict и сетевых интерфейсов
**2.4.3** - Настройка опции загрузки ядра initonalloc
**2.4.4** - Включение опции slab_nomerge
**2.4.5** - Инициализация механизма IOMMU
**2.4.6** - Рандомизация расположения стека ядра
**2.4.7** - Защита от уязвимостей CPU
**2.4.8** - Защита подсистемы eBPF JIT

### reducing_the_attack_perimeter.yaml
**2.5.1** - Отключение устаревшего интерфейса vsyscall
**2.5.2** - Ограничение доступа к событиям производительности
**2.5.3** - Отключение монтирования debugfs
**2.5.4** - Отключение системного вызова kexec_load
**2.5.5** - Ограничение использования пользовательских пространств имен
**2.5.6** - Запрет системного вызова bpf для непривилегированных пользователей
**2.5.7** - Запрет системного вызова userfaultfd
**2.5.8** - Запрет автоматической загрузки модулей ядра
**2.5.9** - Отключение технологии TSX
**2.5.10** - Настройка параметра vm.mmap_min_addr
**2.5.11** - Реализация рандомизации адресного пространства

## Использование

### Базовое использование
```yaml
---
- name: Apply security hardening
  hosts: servers
  become: yes
  
  roles:
    - base/security
```

### С пользовательскими настройками
```yaml
---
- name: Apply custom security settings
  hosts: servers
  become: yes
  
  vars:
    debug_mode: false
    ct_mode: false
    allowed_ssh_subnet:
      - "10.0.0.0/8"
  
  roles:
    - base/security
```

## Требования

- Ansible 2.9+
- Python 3.6+
- Доступ к целевым хостам по SSH с правами sudo
- Поддерживаемые дистрибутивы: Ubuntu 18.04+, CentOS 7+, RHEL 7+

## Зависимости

- `ansible.posix` collection для модуля sysctl

## Примечания

- Многие задачи выполняются только на физических серверах (когда `ct_mode: false`)
- Роль изменяет критические системные параметры - тестируйте в безопасной среде
- Некоторые изменения требуют перезагрузки системы для полного применения
- Рекомендуется создать резервную копию системы перед применением
