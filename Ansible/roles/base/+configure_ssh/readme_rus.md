# SSH Configuration Role - Русская документация

## Обзор

Роль `base.configure_ssh` - это комплексная Ansible роль, предназначенная для настройки SSH серверов с применением лучших практик безопасности и контролем доступа на основе сети. Эта роль обеспечивает детальный контроль над методами аутентификации SSH на основе сетевых адресов источника, гарантируя безопасный удаленный доступ при сохранении операционной гибкости.

## Ключевые возможности

### Контроль доступа на основе сети
- **Доверенные сети**: Разрешить аутентификацию только по паролю для указанных подсетей
- **Недоверенные сети**: Требовать как пароль, так и аутентификацию по публичному ключу
- **Автоматическое управление подсетями**: Извлечение и дедупликация подсетей из определений пользователей
- **Гибкая конфигурация**: Поддержка как прямой конфигурации подсетей, так и конфигурации на основе пользователей

### Усиление безопасности
- Отключен вход root по умолчанию
- Надежные криптографические алгоритмы и шифры
- Ограничение скорости и управление соединениями
- Комплексное логирование и мониторинг
- Конфигурация баннера SSH
- Принудительное использование версии протокола

### Кроссплатформенная поддержка
- **Семейство Debian**: Debian, Ubuntu, Linux Mint
- **Семейство RedHat**: RHEL, CentOS, Rocky Linux, AlmaLinux, Fedora
- **Семейство SUSE**: openSUSE, SLES

### Расширенные возможности
- Комплексная валидация параметров
- Тестирование и валидация конфигурации
- Возможности резервного копирования и восстановления
- Подробный отладочный вывод и логирование
- Управление службами и мониторинг

## Режимы конфигурации

### 1. Прямая конфигурация подсетей

Настройка подсетей напрямую без определений пользователей:

```yaml
- hosts: servers
  roles:
    - role: base.configure_ssh
      vars:
        allowed_subnets:
          - "192.168.1.0/24"
          - "10.20.30.0/24"
        denied_subnets:
          - "192.168.0.0/24"
          - "10.10.10.0/24"
```

### 2. Конфигурация на основе пользователей

Извлечение подсетей из определений пользователей (рекомендуется для интеграции с dashboard):

```yaml
- hosts: servers
  roles:
    - role: base.configure_ssh
      vars:
        users_to_add:
          - username: "admin"
            password: "SecurePassword123!"
            groups: ["sudo", "admin"]
            is_sudoers: true
            shell: /bin/bash
            allowed_subnets:
              - "192.168.1.0/24"
              - "10.20.30.0/24"
            denied_subnets: []
            authorized_keys: []
```

## Переменные

### Конфигурация SSH

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `ssh_port` | `22` | Порт SSH сервера |
| `ssh_protocol` | `"tcp"` | Протокол SSH (tcp/udp) |
| `ssh_service_name` | `"sshd"` | Имя службы SSH |

### Настройки безопасности

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `ssh_security.password_authentication` | `"yes"` | Разрешить аутентификацию по паролю |
| `ssh_security.pubkey_authentication` | `"no"` | Разрешить аутентификацию по публичному ключу |
| `ssh_security.permit_root_login` | `"yes"` | Разрешить вход root |
| `ssh_security.max_auth_tries` | `6` | Максимальное количество попыток аутентификации |
| `ssh_security.login_grace_time` | `60` | Время ожидания входа в секундах |

### Контроль доступа к сети

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `ssh_access_control.trusted_networks_password_auth` | `true` | Разрешить аутентификацию по паролю для доверенных сетей |
| `ssh_access_control.untrusted_networks_require_both` | `true` | Требовать и пароль, и ключ для недоверенных сетей |
| `ssh_access_control.default_network_behavior` | `"restrictive"` | Поведение по умолчанию для неопределенных сетей |

### Расширенные настройки

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `ssh_advanced.x11_forwarding` | `false` | Включить переадресацию X11 |
| `ssh_advanced.allow_tcp_forwarding` | `false` | Разрешить переадресацию TCP |
| `ssh_advanced.use_match_blocks` | `true` | Использовать блоки Match для правил, специфичных для сети |

## Примеры использования

### Базовая конфигурация

```yaml
- hosts: servers
  roles:
    - role: base.configure_ssh
      vars:
        ssh_port: 2222
        users_to_add:
          - username: "admin"
            password: "SecurePassword123!"
            groups: ["sudo"]
            is_sudoers: true
            shell: /bin/bash
            allowed_subnets:
              - "192.168.1.0/24"
            denied_subnets: []
            authorized_keys: []
```

### Расширенная конфигурация безопасности

```yaml
- hosts: servers
  roles:
    - role: base.configure_ssh
      vars:
        ssh_security:
          max_auth_tries: 2
          login_grace_time: 30
          client_alive_interval: 300
          client_alive_count_max: 2
        ssh_advanced:
          x11_forwarding: false
          allow_tcp_forwarding: false
          use_match_blocks: true
        users_to_add:
          - username: "admin"
            password: "VerySecurePassword123!"
            groups: ["sudo", "admin"]
            is_sudoers: true
            shell: /bin/bash
            allowed_subnets:
              - "192.168.1.0/24"
              - "10.20.30.0/24"
            denied_subnets:
              - "192.168.0.0/24"
            authorized_keys:
              - "ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABgQC..."
```

### Интеграция с Dashboard

```yaml
- hosts: dashboard
  roles:
    - role: base.configure_ssh
      vars:
        # Использовать dashboard_users_to_add из host_vars
        users_to_add: "{{ dashboard_users_to_add }}"
        ssh_security:
          password_authentication: "yes"
          pubkey_authentication: "yes"
        ssh_access_control:
          trusted_networks_password_auth: true
          untrusted_networks_require_both: true
```

## Логика контроля доступа к сети

### Доверенные сети
- **Аутентификация по паролю**: Разрешена (если `trusted_networks_password_auth: true`)
- **Аутентификация по публичному ключу**: Всегда разрешена
- **Метод доступа**: Либо пароль, ЛИБО аутентификация по ключу

### Недоверенные сети
- **Аутентификация по паролю**: Разрешена (если `untrusted_networks_require_both: true`)
- **Аутентификация по публичному ключу**: Требуется
- **Метод доступа**: Требуется И пароль, И аутентификация по ключу

### Запрещенные сети
- **Доступ**: Полностью запрещен
- **Реализация**: Использует блоки SSH Match с `DenyUsers *`

## Функции безопасности

### Криптографические настройки
- **Алгоритмы ключей хоста**: RSA-SHA2, ECDSA, Ed25519
- **Алгоритмы KEX**: Curve25519, ECDH, Diffie-Hellman
- **Шифры**: ChaCha20-Poly1305, AES-GCM, AES-CTR
- **MAC**: HMAC-SHA2 с ETM

### Управление соединениями
- **Интервал активности клиента**: 300 секунд
- **Максимальное количество проверок активности клиента**: 2
- **Время ожидания входа**: 60 секунд
- **Максимальное количество попыток аутентификации**: 3

### Логирование и мониторинг
- **Уровень логирования**: INFO
- **Удобство Syslog**: AUTH
- **Баннер**: Предупреждающее сообщение безопасности
- **Последний вход**: Отображается при подключении

## Устранение неполадок

### Распространенные проблемы

1. **SSH служба не запускается**
   - Проверить синтаксис конфигурации SSH: `sshd -t`
   - Проверить имя службы: `systemctl status sshd`
   - Проверить логи: `journalctl -u sshd`

2. **Ошибки аутентификации**
   - Проверить правила контроля доступа к сети
   - Проверить права пользователей и группы
   - Валидировать форматы SSH ключей

3. **Ошибки валидации конфигурации**
   - Включить режим отладки: `debug_mode: true`
   - Проверить результаты валидации параметров
   - Проверить формат подсети (нотация CIDR)

### Режим отладки

Включить режим отладки для подробного вывода:

```yaml
- hosts: servers
  roles:
    - role: base.configure_ssh
      vars:
        debug_mode: true
        validate_configuration: true
```

## Лучшие практики

1. **Использовать конфигурацию на основе пользователей**: Предпочитать `users_to_add` вместо прямой конфигурации подсетей для лучшей интеграции с системами управления пользователями.

2. **Включить валидацию**: Всегда использовать `validate_configuration: true` в производственных средах.

3. **Резервное копирование конфигурации**: Включить `ssh_backup_enabled: true` для автоматических резервных копий.

4. **Тестировать конфигурацию**: Использовать `sshd -t` для валидации конфигурации перед применением изменений.

5. **Мониторить логи**: Регулярно проверять логи SSH на предмет событий безопасности и ошибок аутентификации.

6. **Сегментация сети**: Использовать конкретные подсети для доверенных сетей вместо широких диапазонов.

7. **Управление ключами**: Реализовать правильное управление SSH ключами и политики их ротации.
