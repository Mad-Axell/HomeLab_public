## Описание

Данная Ansible роль создает и управляет учетными записями пользователей в Linux системах с комплексной валидацией, управлением группами и настройкой sudo привилегий. Предоставляет надежное решение для управления пользователями в различных дистрибутивах Linux.

**Ключевые возможности:**
- Создание и обновление учетных записей пользователей
- Хеширование паролей с использованием SHA-512
- Управление группами (создание и назначение)
- Настройка sudo привилегий
- Комплексная валидация входных данных
- Автоматический откат при ошибке
- Проверка после развертывания
- Двуязычный отладочный вывод (английский/русский)

## Возможности

- ✅ **Управление пользователями**: Создание и обновление учетных записей с полной конфигурацией
- ✅ **Управление группами**: Автоматическое создание групп и управление членством в группах
- ✅ **Конфигурация Sudo**: Настройка sudo привилегий с индивидуальными sudoers файлами
- ✅ **Безопасность паролей**: Хеширование паролей SHA-512 с валидацией надежности
- ✅ **Валидация**: Комплексная проверка всех входных данных (имена пользователей, пароли, UID, GID, оболочки, пути)
- ✅ **Предварительные проверки**: Проверка версии Ansible, совместимости ОС и версии Python
- ✅ **Обработка ошибок**: Автоматический откат с функциями резервного копирования/восстановления
- ✅ **Верификация**: Проверки после развертывания для корректной настройки пользователей
- ✅ **Мультиплатформенность**: Поддержка Debian, Ubuntu, RHEL, CentOS, Rocky Linux, openSUSE
- ✅ **Отладочный вывод**: Структурированное логирование с поддержкой английского/русского языков
- ✅ **Зарезервированные имена**: Защита от использования системных зарезервированных имен пользователей

## Переменные роли

### Обязательные переменные

#### `users_to_add`
- **Тип**: Список словарей
- **Обязательно**: Да
- **Описание**: Список пользователей для создания или обновления

Каждый словарь пользователя должен содержать:
- `username` (строка, обязательно): Имя пользователя для учетной записи
- `password` (строка, обязательно): Пароль открытым текстом (будет автоматически захеширован)

Пример:
```yaml
users_to_add:
  - username: admin
    password: "БезопасныйПароль123"
```

### Опциональные переменные

#### Конфигурация отладки

| Переменная | Тип | По умолчанию | Описание |
|-----------|-----|--------------|----------|
| `debug_mode` | логический | `false` | Включить детальный вывод отладки |
| `debug_lang` | строка | `both` | Язык отладки: `english`, `russian` или `both` |
| `debug_show_passwords` | логический | `false` | Показывать пароли в отладочном выводе (НЕБЕЗОПАСНО - только для тестирования) |

#### Настройки пользователей по умолчанию

| Переменная | Тип | По умолчанию | Описание |
|-----------|-----|--------------|----------|
| `add_users_create_home` | логический | `true` | Создавать домашнюю директорию по умолчанию |
| `add_users_home_prefix` | строка | `/home` | Префикс пути для домашних директорий пользователей |
| `add_users_default_shell` | строка | `/bin/bash` | Оболочка по умолчанию для пользователей |

#### Валидация паролей

| Переменная | Тип | По умолчанию | Описание |
|-----------|-----|--------------|----------|
| `add_users_validate_passwords` | логический | `true` | Включить валидацию надежности паролей |
| `add_users_min_password_length` | целое число | `8` | Требуемая минимальная длина пароля |

#### Откат и резервное копирование

| Переменная | Тип | По умолчанию | Описание |
|-----------|-----|--------------|----------|
| `enable_rollback` | логический | `true` | Включить автоматический откат при ошибке |
| `backup_config` | логический | `true` | Создавать резервную копию перед изменениями |

#### Производительность

| Переменная | Тип | По умолчанию | Описание |
|-----------|-----|--------------|----------|
| `retries` | целое число | `3` | Количество повторов для неудачных задач |
| `retry_delay` | целое число | `5` | Задержка между повторами в секундах |

#### Верификация

| Переменная | Тип | По умолчанию | Описание |
|-----------|-----|--------------|----------|
| `verify_deployment` | логический | `true` | Включить проверку после развертывания |
| `verify_user_login` | логический | `true` | Проверить возможность входа пользователей |

### Параметры конфигурации пользователя

Каждый пользователь в списке `users_to_add` поддерживает следующие опции:

| Опция | Тип | Обязательно | По умолчанию | Описание |
|-------|-----|-------------|--------------|----------|
| `username` | строка | Да | - | Имя пользователя (только строчные буквы, цифры, подчеркивания) |
| `password` | строка | Да | - | Пароль открытым текстом (будет захеширован) |
| `groups` | список | Нет | `[]` | Список групп, в которых должен состоять пользователь |
| `is_sudoers` | логический | Нет | `false` | Предоставить sudo привилегии |
| `shell` | строка | Нет | `/bin/bash` | Оболочка пользователя (должна быть в списке valid_shells) |
| `uid` | целое число | Нет | авто | ID пользователя (1000-60000 для обычных пользователей) |
| `gid` | целое число | Нет | авто | ID основной группы |
| `create_home` | логический | Нет | `true` | Создать домашнюю директорию |
| `home` | строка | Нет | `/home/{username}` | Пользовательский путь домашней директории (должен быть абсолютным) |

## Примеры плейбуков

### Базовое использование

Создание одного пользователя с sudo привилегиями:

```yaml
---
- name: Создание администратора
  hosts: all
  become: true
  
  vars:
    users_to_add:
      - username: admin
        password: "БезопасныйПароль123"
        is_sudoers: true
        
  roles:
    - role: base.add_users
```

### Расширенное использование

Создание нескольких пользователей с различными конфигурациями:

```yaml
---
- name: Настройка пользователей команды разработки
  hosts: development_servers
  become: true
  
  vars:
    debug_mode: true
    debug_lang: russian
    add_users_min_password_length: 12
    
    users_to_add:
      # Системный администратор
      - username: sysadmin
        password: "Admin!БезопПароль2024"
        groups: ["docker", "wheel", "systemd-journal"]
        is_sudoers: true
        shell: /bin/bash
        
      # Разработчик с определенным UID
      - username: developer
        password: "Dev!БезопПароль2024"
        groups: ["docker", "git", "developers"]
        is_sudoers: true
        shell: /bin/zsh
        uid: 5000
        
      # Пользователь только для чтения (мониторинг)
      - username: monitoring
        password: "Monitor!Пароль2024"
        groups: ["systemd-journal"]
        is_sudoers: false
        shell: /bin/sh
        
      # Сервисная учетная запись
      - username: appuser
        password: "App!СервисПароль2024"
        groups: ["app"]
        is_sudoers: false
        create_home: false
        home: /opt/app

  roles:
    - role: base.add_users
      tags: ['users', 'setup']
```

### Использование в продакшене

Развертывание в продакшене с полной обработкой ошибок:

```yaml
---
- name: Управление пользователями в продакшене
  hosts: production_servers
  become: true
  serial: 5  # Обрабатывать по 5 серверов за раз
  max_fail_percentage: 10
  
  vars:
    # Настройки для продакшена
    debug_mode: false
    add_users_validate_passwords: true
    add_users_min_password_length: 16
    enable_rollback: true
    backup_config: true
    verify_deployment: true
    
    # Загрузка пользователей из vault
    users_to_add: "{{ vault_production_users }}"
    
  pre_tasks:
    - name: Убедиться что мы на продакшн серверах
      ansible.builtin.assert:
        that:
          - inventory_hostname in groups['production']
          - ansible_distribution in ['Ubuntu', 'Debian', 'RedHat', 'Rocky']
        fail_msg: "Этот плейбук должен запускаться только на продакшн серверах"
        
  roles:
    - role: base.add_users
      tags: ['users', 'security']
      
  post_tasks:
    - name: Отправка уведомления
      ansible.builtin.debug:
        msg: "Управление пользователями завершено на {{ inventory_hostname }}"
```

## Обработка ошибок

Роль реализует комплексную обработку ошибок:

### Автоматический откат
Когда `enable_rollback: true` (по умолчанию), роль будет:
1. Создавать резервные копии перед внесением изменений
2. Валидировать все конфигурации
3. Автоматически восстанавливать предыдущее состояние при ошибке
4. Предоставлять подробные сообщения об ошибках

### Стратегия резервного копирования
Когда `backup_config: true` (по умолчанию):
- Sudoers файлы резервируются в `/root/sudoers_backup_<timestamp>.tar.gz`
- Индивидуальные sudoers файлы резервируются перед изменением
- Откат использует эти резервные копии для восстановления предыдущего состояния

### Ошибки валидации
Роль проверяет:
- Формат имени пользователя (только строчные, цифры, подчеркивания)
- Длину пароля (настраиваемый минимум)
- Зарезервированные системные имена пользователей
- Диапазоны UID/GID
- Пути оболочек (должны быть в списке valid_shells)
- Пути домашних директорий (должны быть абсолютными)
- Дублирующиеся имена пользователей в конфигурации

### Механизм повторов
Неудачные задачи автоматически повторяются с настраиваемыми параметрами:
- Повторов по умолчанию: 3
- Задержка по умолчанию: 5 секунд
- Настраивается через переменные `retries` и `retry_delay`

## Функции безопасности

### Безопасность паролей
- **Хеширование**: Алгоритм SHA-512 для всех паролей
- **Валидация**: Настраиваемые требования минимальной длины
- **Без логирования**: Все операции с паролями используют `no_log: true`
- **Безопасность отладки**: Пароли скрыты в отладочном выводе, если не включено явно

### Права доступа к файлам
- Sudoers файлы: `0440` (только чтение для root и группы sudo)
- Shadow файлы: `0000` (только root)
- Конфигурационные файлы: `0644` (чтение для всех, запись для владельца)
- Домашние директории: `0755` (доступ для пользователя)

### Контроль доступа
- Защита зарезервированных имен пользователей (root, daemon, bin, sys, и т.д.)
- Валидация диапазона UID (1000-60000 для обычных пользователей)
- Управление sudo привилегиями с индивидуальными sudoers файлами
- Контроль доступа на основе групп

### Валидация
- Валидация синтаксиса sudoers файлов с помощью `visudo -c`
- Принудительное форматирование имени пользователя
- Валидация пути оболочки
- Валидация пути домашней директории

## Верификация

Роль включает проверку после развертывания (когда `verify_deployment: true`):

### Выполняемые проверки
1. ✅ Пользователь существует в базе данных passwd
2. ✅ Членство пользователя в группах корректно
3. ✅ Sudoers файлы существуют для sudo пользователей
4. ✅ Sudoers файлы отсутствуют для обычных пользователей
5. ✅ Домашние директории существуют (когда create_home: true)
6. ✅ Конфигурация оболочки пользователя корректна

### Вывод верификации
Когда `debug_mode: true`, отображаются детальные результаты верификации:
- Всего пользователей проверено
- Успешно созданных пользователей
- Проверенных sudo пользователей
- Созданных домашних директорий
- Общий статус верификации

## Устранение неполадок

### Распространенные проблемы

**Проблема**: Создание пользователя не удается с ошибкой "Имя пользователя содержит недопустимые символы"
```
Решение: Используйте только строчные буквы, цифры и подчеркивания в именах пользователей
Допустимо: admin, user_1, dev123
Недопустимо: Admin, user-1, dev.user
```

**Проблема**: Валидация пароля не проходит
```
Решение: Убедитесь, что пароль соответствует требованиям минимальной длины
- Проверьте настройку add_users_min_password_length
- Минимум по умолчанию: 8 символов
- Используйте надежные пароли с разным регистром, цифрами, символами
```

**Проблема**: Конфигурация sudo не проходит
```
Решение: Проверьте синтаксис sudoers файла
- Проверьте логи на наличие ошибок валидации
- Убедитесь, что backup_config: true для безопасности
- Ручная валидация: sudo visudo -c
```

**Проблема**: Роль не работает на неподдерживаемой ОС
```
Решение: Проверьте список поддерживаемых ОС
- Ubuntu 20.04+, Debian 11+, RHEL/CentOS/Rocky 8+, openSUSE 15+
- Обновите ОС или используйте совместимую версию
```

### Режим отладки

Включите детальную отладку:
```yaml
vars:
  debug_mode: true
  debug_lang: russian
  debug_show_passwords: false  # Установите true только в тестовых окружениях
```

Отладочный вывод включает:
- Детали конфигурации пользователя
- Системную информацию
- Результаты создания групп
- Результаты создания/обновления пользователей
- Статус конфигурации sudo
- Результаты верификации

### Файлы логов

Проверьте системные логи на наличие ошибок:
```bash
# Просмотр логов выполнения роли
journalctl -xe

# Проверка логов аутентификации
tail -f /var/log/auth.log  # Debian/Ubuntu
tail -f /var/log/secure    # RHEL/CentOS
```
